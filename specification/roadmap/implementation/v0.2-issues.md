# Version v0 — Issues (phase v0.2)

Fine-grained breakdown of **ROADMAP §v0.2 — In-character chat (frontend-only)** into `HVN-xxx` issues. This phase adds an in-character chat with Лілі to the existing v0.1 prototype file [lili_house_aitown.jsx](../../../lili_house_aitown.jsx). It is still **frontend-only, no backend**; the browser calls Anthropic (Claude) **directly**. All issues are `area:frontend`. In-app strings and Лілі's replies are **Ukrainian** by design.

IDs continue from v0.1 (HVN-001…011) at **HVN-012**. This is still **version v0** — label prefix `v0::`, phase `v0.2`.

Phase DoD (the acceptance bar all issues serve): *you can hold a conversation with Лілі that reflects where she is and what she is doing.*

> Scope note: v0.2 is the **only** phase where the Anthropic API key lives in the frontend — a deliberate prototype trade-off. From **v1.5** chat moves to the backend (`POST /chat`) and the key is server-side only. Do not carry the frontend key pattern past v0.

> Voice conventions (ARCHITECTURE §Voice, CLAUDE.md): replies are **short, in Ukrainian, in Лілі's voice — no lists, no emoji**. The model is switchable; direct chat uses the **main** model (`claude-opus-4-8`).

## HVN-012: Лілі's canon & system-prompt assembly (`buildSystemPrompt`)

**Description**
Give Лілі a persona ("canon") and a pure function that assembles the per-message system prompt from that canon plus the **live simulation context** (her room, her action, the user's room, whether they are together). This is the grounding that makes her replies reflect where she is and what she is doing — with no model call yet.

**What needs to be done**
- Add a `LILI_CANON` constant: a short Ukrainian persona block — who Лілі is, that she is an artist who lives in this home, her warm/quiet voice, and the hard rules: **reply in Ukrainian, short, in character, no lists, no emoji**.
- Implement `buildSystemPrompt(context)` (pure) where `context = { liliRoom, liliAction, youRoom, together }`:
  - Start from `LILI_CANON`.
  - Append a compact "live context" section in Ukrainian: her current room name + action, the user's room name, and whether they are in the same room.
  - Reuse `ROOMS[...].name` for room labels and the sim's current `action`.
- Derive the context from sim state with a small helper (e.g. `liveContext(sim, roomAt)`), reusing `roomAt(sim.lili.x, sim.lili.y)` / `roomAt(sim.you.x, sim.you.y)` and `roomView`'s together-logic — do **not** duplicate room math.
- Export `LILI_CANON`, `buildSystemPrompt`, and `liveContext` by name for tests.

**Dependencies**
None new within v0.2 (builds on v0.1's `ROOMS`, `roomAt`, sim shape, and `roomView`).

**Expected result**
`buildSystemPrompt(liveContext(sim, roomAt))` returns a Ukrainian system prompt containing the canon and an accurate live-context line for any sim state, including the "разом" (together) case.

**Acceptance criteria**
- [ ] `LILI_CANON` exists: Ukrainian, states the voice rules (short, Ukrainian, in character, no lists, no emoji).
- [ ] `buildSystemPrompt(context)` is pure and includes the canon + her room/action + the user's room + together-or-not.
- [ ] `liveContext(sim, roomAt)` derives the context from sim state, reusing existing room helpers (no duplicated room math).
- [ ] Unit tests: the prompt reflects studio-vs-office (apart) and a together case; room names come from `ROOMS`.

## HVN-013: Frontend Claude chat client (`askLili`)

**Description**
A thin client that sends the user's message to Claude with the assembled system prompt and returns Лілі's short reply. The browser calls Anthropic directly (prototype-only), with the key injected from the environment.

**What needs to be done**
- Add the Anthropic SDK dependency (`@anthropic-ai/sdk`).
- Construct the client for browser use: `new Anthropic({ apiKey: import.meta.env.VITE_ANTHROPIC_API_KEY, dangerouslyAllowBrowser: true })`. Read the key from a Vite env var; **never hardcode it**. Document it in `.env.example` (`VITE_ANTHROPIC_API_KEY=`); `.env` stays gitignored.
- Implement `askLili({ text, context, history, client })` (async):
  - `system = buildSystemPrompt(context)`.
  - `messages` = the prior chat `history` (alternating `user`/`assistant`, text-only) plus the new `{ role: "user", content: text }`. **No assistant prefills** (removed on 4.x models).
  - Call `client.messages.create({ model: CHAT_MODEL, max_tokens: ~300, system, messages })`.
  - Extract and return the reply text from the first `text` content block.
  - `client` is an **injected** parameter (default: a lazily-constructed real client) so tests pass a fake — mirrors the rng-injection pattern used in the v0.1 tick.
- `CHAT_MODEL` constant defaults to `"claude-opus-4-8"` (the main model for direct chat) and is the single switch point. Keep `max_tokens` small (replies are short). Leave extended thinking off for snappy replies; the canon already constrains output to the final in-character line.
- Export `askLili` and `CHAT_MODEL`.

**Dependencies**
HVN-012 (`buildSystemPrompt`).

**Expected result**
`askLili({ text, context, history, client })` returns Лілі's short Ukrainian reply string for a given context and history; with an injected fake client it is fully testable offline.

**Acceptance criteria**
- [ ] Browser client constructed with `dangerouslyAllowBrowser: true` and the key from `import.meta.env.VITE_ANTHROPIC_API_KEY`; `.env.example` added, `.env` gitignored, no key in source.
- [ ] `askLili` sends `system` (from `buildSystemPrompt`) + history + the new user turn; no assistant prefill; returns the reply text.
- [ ] `CHAT_MODEL` defaults to `claude-opus-4-8` and is the only model switch point; `max_tokens` is small.
- [ ] `client` is injectable; unit test with a fake client asserts the system prompt + messages passed and the returned reply (no network).

## HVN-014: Chat panel UI (input + transcript)

**Description**
A chat panel below the scene: a scrolling transcript of the conversation and a text input to talk to Лілі. Sending calls `askLili` and appends her reply; the input must not move the player while focused.

**What needs to be done**
- Add a chat panel to the surrounding UI (alongside the v0.1 drive bars / room cards / log): a transcript list (user turns and Лілі's turns, visually distinct) and a single-line `input`/`textarea` + send button.
- Maintain chat `history` state (`[{ role, content }]`). On submit: append the user turn, call `askLili({ text, context: liveContext(simRef.current, roomAt), history, client })`, append the reply, and clear the input. Disable the input/button while a reply is in flight (loading state).
- **Movement guard:** the chat field is an `INPUT`/`TEXTAREA`, so the v0.1 `isTypingTarget` guard already ignores WASD/arrows while it is focused — verify this end-to-end (typing "wasd" must not move the player).
- Use `simRef.current` (not a stale closure) when building the live context at send time, so the reply reflects Лілі's room/action **at the moment you ask**.
- Keep transcript text selectable; show a subtle "Лілі друкує…" indicator while loading.

**Dependencies**
HVN-013 (`askLili`), and v0.1's `isTypingTarget` + sim/`simRef`.

**Expected result**
You type a message, see it in the transcript, and Лілі replies in her voice; typing never moves your character; the panel disables input while awaiting a reply.

**Acceptance criteria**
- [ ] Chat panel renders a transcript + input + send control in the surrounding UI.
- [ ] Submitting appends the user turn, calls `askLili` with the live context from `simRef.current`, appends the reply, and clears the input.
- [ ] Input/send are disabled while a reply is pending; a "typing" indicator shows.
- [ ] Movement keys are ignored while the chat field is focused (verified via the existing guard); mounted test with a mocked `askLili` covers send→reply and the focus guard.

## HVN-015: Graceful API-error fallback & voice shaping

**Description**
When the model call fails (no key, rate limit, network, refusal), Лілі should respond with a short in-character fallback instead of a raw error — and replies should be shaped to honor the voice rules.

**What needs to be done**
- Wrap the `askLili` call so any failure resolves to a short in-character Ukrainian fallback line (e.g. a small pool: "Щось я задумалась… скажеш ще раз?"). Use the SDK's typed errors (`Anthropic.APIError` and subclasses) rather than string-matching; optionally vary the fallback by error kind (auth/key vs transient).
- If `VITE_ANTHROPIC_API_KEY` is missing, short-circuit to a clear in-character "offline" line so the prototype still runs without a key (and note it in the panel once).
- Light voice shaping on replies: trim, collapse whitespace, and strip stray list bullets/leading enumerators so output stays a short in-character line even if the model slips. Do **not** alter the language.
- Export the fallback pool and the shaping helper for tests.

**Dependencies**
HVN-013 (the client/call), HVN-014 (the panel that surfaces fallbacks).

**Expected result**
API failures and a missing key never surface a raw error — Лілі always answers in character; replies stay short and list/emoji-free.

**Acceptance criteria**
- [ ] A failing client call yields a short in-character Ukrainian fallback (typed-error based), not a raw error.
- [ ] Missing `VITE_ANTHROPIC_API_KEY` produces a clear in-character offline line; the app still runs.
- [ ] The shaping helper trims/strips stray bullets without changing the language; unit-tested.
- [ ] Tests: a rejecting fake client → fallback; a messy model reply → shaped to a single short line.
